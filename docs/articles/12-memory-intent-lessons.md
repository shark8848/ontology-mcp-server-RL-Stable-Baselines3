# 记忆驱动的意图识别与推理闭环：经验与教训

> **阅读顺序提示**：架构细节、数据流和 API 请参考《03-ChromaDB + MCP：支撑多轮上下文记忆与工具编排的语义骨架》。本文只关心那些在实战中踩过的坑与总结出的工作原则，帮助团队把“知道怎么做”升级为“知道为何这么做、做错会怎样”。

## 1. 记忆如何重塑意图识别（真实收益版）

| 经验做法 | 实测影响 | 量化改观 |
| --- | --- | --- |
| 只把最近 5 轮摘要 + 用户档案打包进 prompt，拒绝“全量回放” | LLM 不再把“确认支付”“继续推荐”判成闲聊 | checkout/recommendation 场景 embedding 置信度 +12%（0.83→0.95） |
| 工具一执行就写入 `UserContextManager`（手机号、地址、product_id） | “请再给地址”式追问几乎消失 | 相关会话平均少 1.4 轮 |
| IntentTracker 先读记忆标签再调用 LLM 分类 | 过滤掉不可能的类，避免上下文跳闸 | 售后场景误判率从 21% 降到 4% |

**Lesson #1**：意图识别并不是装一个分类器这么简单；真正可靠的 pipeline = 记忆感知 + Prompt 约束 + 轻量分类。缺任意一环都会退化成“零记忆 ChatGPT”。

## 2. 推理链条的正向增益（以及踩过的坑）

1. **Prompt 补完**：在 `_build_user_message` 里拼“上下文摘要 + 当前问题”，LLM 能直接引用订单号、库存结果。若忘记过滤旧的“图表不可用”提示，会导致 LLM 一直道歉——我们通过 `_filter_negative_history` 解决。  
2. **验证守卫依赖记忆**：`_should_require_validation` 需要读取 `tool_log` 才能判断是否必须跑 SHACL。早期忘了写入确认分支的 order 记录，导致 guard 误以为“尚未下单”，从而放走支付流程。  
3. **确认模式闭环**：直到我们把确认执行的工具补写回 `tool_log`/memory/state，LLM 才终于知道订单已建好，下一句“确认支付”才会直达支付，而非重复 `commerce_create_order`。

> **教训**：记忆一致性比容量更重要——只要有一条执行路径没写记忆，下一轮就会“失忆”并重复推理，甚至触发死循环。

## 3. 高频事故清单与快速止血

| 风险 | 症状 | 缓解措施 |
| --- | --- | --- |
| 负面历史污染 | “图表不可用”之类旧消息被 LLM 循环引用 | `_filter_negative_history` 过滤 + 重写摘要模板 |
| 记忆爆炸 | prompt 超 8k token、成本暴涨 | 严控 recent_turns=5 + 语义检索片段，结构化字段单独注入 |
| 确认路径失忆 | 订单确认后一问再问“是否要下单” | `confirmation_mode` 中补写 `tool_log`、state、memory |
| 语义检索误命中 | 过往 VIP 对话影响新用户推荐 | metadata 加 `user_id/session_id` 条件，检索时强制匹配 |


## 4. 结论：记忆是推理内核，而非附属品

- 记忆不是锦上添花，而是电商智能体完成闭环推理的地基。  
- 意图识别准确率 ≈ 记忆质量 × prompt 设计 × 分类器稳定性。  
- 任何绕开主循环的执行路径，都必须显式写回记忆与状态，否则下一轮会在关键节点“断片”。

通过这些经验和教训，我们把记忆系统从“可选组件”升级成“推理内核”，也让团队在后续需求评审时有了判断依据：凡是影响意图识别或推理链条的改动，都必须评估对记忆一致性的影响。
